**Step 4 — Score each candidate across four dimensions (0-100 each):**

**4a. Capability Match (weight: 0.40):**
Compare candidate `name` + `description` against the user's original task query (from state). Score based on keyword overlap and semantic relevance:
- Extract keywords from user query (split on spaces, remove stopwords: the, a, an, for, to, of, in, and, or, with)
- Count keyword hits in candidate name + description (case-insensitive)
- Score = min(100, (hits / total_keywords) * 100)
- If 0 keywords match, score = 0

**4b. Security Posture (weight: 0.25):**
- Use `security_score` directly if present (already 0-100)
- If null, default to 50 (unknown = neutral)
- If security_score < 30: set `security_warning: true` on candidate

**4c. Popularity (weight: 0.20):**
- Normalize: `pop_score = min(100, ((stars || 0) * 2 + (install_count || 0) / 10))`
- Cap at 100. If both null, score = 0.

**4d. Recency (weight: 0.15):**
- Parse `last_updated` as ISO 8601 date
- days_ago = (now - last_updated) in days
- If days_ago <= 30: score = 100
- If days_ago <= 90: score = 80
- If days_ago <= 365: score = 50
- If days_ago > 365: score = 20
- If null: score = 10

**Step 5 — Compute weighted composite:**
```
composite = (capability * 0.40) + (security * 0.25) + (popularity * 0.20) + (recency * 0.15)
```
Round to 2 decimal places.

**Determinism guarantee:** Sort candidates by composite DESC. On tie, sort by name ASC (lexicographic). This ensures identical input always produces identical ranking.

**Security warning:** Any candidate with security_score < 30 gets `security_warning: true` in output. Log: `"SECURITY WARNING: {name} has security_score {score} (< 30 threshold)"`
