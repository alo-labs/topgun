## practice:enforcement

2026-04-18 — LLM instruction compliance has a ceiling: prompt instructions are suggestions, not guarantees. A model that ignores an instruction once will ignore a stronger version of the same instruction. The only durable enforcement is mechanical: move the decision outside the LLM's control path (subprocess, binary, OS hook).

2026-04-18 — Hook layering for critical invariants: for any invariant that must hold unconditionally, implement three layers — (1) prompt instruction for intent communication, (2) binary self-check for surfacing failures before they propagate, (3) OS-level hook (PreToolUse/PostToolUse) as the only layer that cannot be bypassed by agent behavior. Design the system to degrade gracefully when layers 1 and 2 are bypassed but layer 3 holds.

## stack:claude-code-plugins

2026-04-18 — Plugin cache is immutable during development: `~/.claude/plugins/cache/` is typically blocked by Silver Bullet's dev-cycle-check hook. Plugin changes cannot be synced directly to the cache during a session. The only path is git commit → version tag → GitHub release → user reinstall from marketplace. Factor this into the release workflow.

2026-04-18 — `--bare` subprocesses for speed: spawning `claude --bare` skips hooks, LSP, CLAUDE.md auto-discovery, and plugin sync. Useful for adapter subprocesses that need to be fast and isolated. The tradeoff: prompts must be fully self-contained since no project context is loaded.

2026-04-18 — Plugin manifest version alignment: any plugin with two version-bearing manifest files (e.g., plugin.json + marketplace.json) must keep them in sync. Mismatch causes silent failures (skills/agents not exposed) with no error message. Automate the check in CI or a pre-release gate.

## devops:releases

2026-04-18 — Tag deletion and recreation breaks downstream caches: deleting and re-pushing a git tag (e.g., v1.2.0) causes any system that cached the old tag's SHA to serve stale content. Always prefer creating a new patch version (v1.2.1) rather than retagging. Only retag if the release has zero downstream installs.

## practice:parallel-dispatch

2026-04-18 — Partial-file aggregation pattern for parallel fan-out: when spawning N parallel workers, have each worker write its own output file before a coordinator aggregates. Workers that fail write an explicit `{ "status": "unavailable" }` file rather than omitting output. The coordinator counts files rather than trusting a success/failure signal. This makes failures countable, auditable, and explicit — and lets the coordinator distinguish "worker failed" from "worker never started".
